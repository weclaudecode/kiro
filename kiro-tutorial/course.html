What an LLM actually is

Before we touch kiro-cli, we have to break a few assumptions. You've worked with stateful services, deterministic functions, and well-typed APIs for years. LLMs work very differently from what your intuition expects.

1.1 The mental model: next-token prediction

An LLM is, at its core, a function:

f(text_in) -> probability_distribution_over_next_token

Given some input text, it outputs a probability for every possible next token. It samples one. Appends. Repeats. That's it.

It is not "reasoning" in the human sense. It is not consulting a database. It is not retrieving facts. It is pattern-matching token sequences based on its training data and what it has seen in the current input.

Analogy You Already Know

Autocomplete on your phone, scaled massively. Your phone predicts the next word from 2–3 recent ones. An LLM predicts the next token from up to ~200,000 tokens of context.

Every response from kiro-cli is the model generating tokens one at a time based on whatever you put in front of it. The discipline of agentic coding is what you put in front of it.

1.2 Stateless and single-turn

LLMs have no memory between calls. None.

Closest analogy from your world: HTTP. Every request must carry everything the server needs to respond. The server has no idea you called it five minutes ago.

LLM calls work the same way:

# Call 1
user: "What's the capital of France?"
       -> "Paris."

# Call 2 (no history replayed)
user: "What did I just ask?"
       -> [model has no idea]

For call 2 to work, the client must replay the history:

# Call 2 (with history replayed)
[
  user:      "What's the capital of France?",
  assistant: "Paris.",
  user:      "What did I just ask?"
]
       -> "You asked about the capital of France."

The illusion of memory in a chat session is achieved by the client replaying all prior messages on every turn. This isn't a kiro-cli quirk — it is how every LLM-backed chat works.

Engineering Implication
  • The cost of your conversation grows linearly with history. By turn 50, you ship 50 turns of history per request.
  • "Forgetting" is the default. Persistence has to be engineered — steering files, indexed knowledge, MCP servers (Modules 3-5).

1.3 Non-determinism: the deal-breaker for engineers

LLMs sample from a probability distribution. Given the same input, you may get different outputs. This is by design.

A few knobs control it:

KnobEffect
temperatureHigher = more randomness. 0 = greedy (always pick the most likely token). 1+ = exploratory.
top_p, top_kRestrict the sampling pool to the most probable tokens.
seedSome APIs accept it. Not all providers honour it strictly.

Even at temperature: 0 you are not guaranteed perfect determinism. Floating-point ordering, batching effects on inference servers, and provider-side caching all break it. Treat LLMs as non-deterministic by default.

If your workflow assumes "same input -> same output", it will break.

Three things this means in practice:

  1. Don't pin behavior with a single example prompt. Pin it with rules, schemas, and validations (steering, hooks, tests).
  2. Treat agent output like untrusted user input — validate, lint, run tests, never auto-merge.
  3. Reproducibility comes from constraining the input (system prompt, tools, schemas) and gating the output (tests, reviews, hooks). It does not come from the model itself.
Why This Matters For The Whole Course

This single property — non-determinism — is why every other primitive in kiro-cli exists. Steering, agents, skills, hooks, specs. They are all tools for constraining input and gating output to make a non-deterministic engine produce reliable results.

1.4 Context window: the working set

The context window is the maximum number of tokens the model can see in one call. Today's frontier models: up to 200K tokens, roughly 500 pages of code/text.

Rough conversions:

  • 1 token ≈ 0.75 English words, or 3–4 characters of code
  • A typical Python file: 500–2,000 tokens
  • A medium MR diff: 1,000–5,000 tokens
  • Your whole repo: 500K–5M+ tokens — will not fit

When you exceed the window, the request either fails or older content is silently truncated depending on the client. kiro-cli does not magically fit your whole repo into context. It chooses what to include — and your steering files, skills, and @-references in chat are how you guide those choices.

This is why context management is the single most important skill in agentic coding (Module 10).

1.5 Knowledge cutoff and hallucination

The model is trained on data up to a certain date (its knowledge cutoff). It does not know what happened after. Anything it "knows" is frozen at training time.

When asked about things it doesn't know, it sometimes generates plausible-sounding but fabricated content. This is hallucination. It is not a bug. It is a fundamental property of pattern-matching token predictors filling in gaps with the most probable continuation.

Counter-Measures (Preview of Later Modules)
  • Grounding — give the model real data via tool calls: file reads, MCP servers, web fetches (Modules 2, 3).
  • Forced verification — instruct the agent to check the actual file or source before answering (Module 4 — steering).
  • Output constraints — schemas, code that must compile, tests that must pass. Fabrications fail loudly (Modules 8, 9 — hooks, specs).

Why MCP exists (preview): to give the model trustworthy, standardized ways to ground itself in real-world data instead of pattern-matching its way through hallucinations.

Run-the-same-prompt-three-times

In kiro-cli, run this exact prompt three times in three fresh sessions:

What are 5 unusual TypeScript features that surprise senior engineers?

You will get three different lists. Some items will overlap. Some will not.

  1. Which items appear all three times?  strong signal, likely real
  2. Which appear once?  weaker signal, possibly hallucinated
  3. How would you verify the weaker ones?

Tool calls into the TypeScript docs via MCP. We'll wire this up in Module 3. The takeaway: never trust a single LLM response on a factual claim. Reproduce, cross-check, or ground it.

This drives home all five sections at once: stateless (fresh sessions), non-deterministic (different outputs), hallucination risk (the singletons), and the need for grounding (verification via real sources).

From chat to agent

A chat answers. An agent acts. The difference is two things: tools and a loop. Once you grasp the loop, every kiro-cli feature stops feeling like magic and starts looking like deliberate engineering.

2.1 The leap: from text-out to action-out

In Module 1 we saw the LLM is a function: text in, text out. By itself, it cannot:

  • Read a file
  • Run a command
  • Query a database
  • Hit an API
  • Even know the current time

Everything it "knows" is frozen at training time. Everything it can affect in the real world is nothing.

The leap to agent is allowing the LLM to emit structured intents — "I want to call this tool with these arguments" — that a surrounding runtime executes on its behalf. The result is fed back into the conversation as if it were a new user turn. The model continues.

"Agentic" = LLM + tools + a loop that mediates them.

2.2 Tool calling: an RPC the model writes itself

Modern LLMs are trained to recognize "I should use a tool" and emit a structured request:

{
  "tool": "fs_read",
  "input": { "path": "src/auth.py" }
}

The agent harness — kiro-cli in our case — sees this, calls the actual fs_read function, and feeds the result back to the model on the next turn:

{
  "tool_result": "def authenticate(token): ...",
  "tool_call_id": "abc123"
}

The model now sees the file contents and can reason about them.

Tool calling is an RPC where the LLM is the client and writes the call site at runtime.

You don't write agent.fs_read("src/auth.py") in your code. The LLM decides to. Your job is to (a) tell it which tools exist and (b) constrain when it should use them.

2.3 The agent loop

Here is the loop in pseudocode — essentially what kiro-cli runs:

while True:
    response = llm.complete(messages, tools=available_tools)

    if response.is_tool_call:
        result = execute_tool(response.tool, response.input)
        messages.append(response)
        messages.append(result)
        continue                     # go around again

    else:
        return response.text         # final answer to user

Four properties matter:

  1. Multi-step by default. One user prompt can trigger many LLM turns, each with one or more tool calls, before a final answer.
  2. The model decides when to stop. It returns plain text when it judges the task done.
  3. The harness mediates everything. Tool whitelists, hooks, error handling, progress reporting.
  4. State lives in messages. The model is still stateless; the harness rebuilds the message list each turn.

This loop is the unit of agentic work. Everything else — subagents, hooks, specs — is structure layered on top.

2.4 Why one prompt isn't enough

Real engineering work is multi-step. Example: "Fix the failing test in auth.py."

TurnModel action
1fs_read("tests/test_auth.py")
2fs_read("src/auth.py")
3execute_bash("pytest tests/test_auth.py -v") to see failure
4Reason about the failure (no tool call)
5fs_write("src/auth.py", <fixed code>)
6execute_bash("pytest tests/test_auth.py") to verify
7Return: "Fixed the bug — verify_signature was returning before checking expiry."

Seven turns. One user prompt. This is what "agent" buys you. A chat model could only have answered turn 4. It couldn't have read the files, run the tests, or written the fix.

2.5 The harness: kiro-cli's actual job

The agent harness is the program around the LLM that makes the loop possible. kiro-cli is a harness. Its core jobs:

  • Tool registry — which tools are available, what their schemas look like
  • Tool execution — actually running fs_read, execute_bash, MCP calls
  • Safety layer — permission prompts, denied commands, allowedPaths, hooks
  • Context management — what messages get sent on each turn (Module 10)
  • Subagent orchestration — spawning child agents (Module 7)
  • User interface — TUI, slash commands, headless mode (Module 12)

Built-in tools in kiro-cli (preview):

ToolWhat it does
fs_readRead a file
fs_writeWrite or edit a file
execute_bashRun shell commands
web_search / web_fetchSearch and fetch URLs
subagentSpawn a child agent (Module 7)
report_progressSurface status to the user
MCP toolsAnything from connected MCP servers (Module 3)
The harness is the substrate. The LLM is the brain. Tools are the hands. Steering, agents, skills, hooks, and specs are the rules of engagement.

2.6 What stops an agent running forever?

A reasonable thing to ask early. Three forces:

  1. The model decides. When it judges the task complete, it returns plain text instead of another tool call. Primary mechanism.
  2. Safety limits in the harness. Max iterations, max tokens, manual Ctrl+C. kiro-cli has these.
  3. Hooks. You can write a stop hook that gates termination (Module 8).

In practice, a well-prompted agent on a well-scoped task stops in 5-30 turns. A poorly-scoped agent on an open-ended task can loop until you stop it. Scoping work and giving clear stopping criteria is part of the skill.

Watch the loop run

Open kiro-cli in a real repo. Ask:

What stack does this project use? Just read the relevant config files
and give me a one-paragraph summary.

Watch the tool calls. You should see something like:

  • fs_read("package.json") or fs_read("pyproject.toml") or fs_read("requirements.txt")
  • Possibly fs_read("README.md")
  • A final text response with no tool call

Count the turns. Note: the model did not answer until it had grounded itself in real files. That grounding behavior comes from training plus kiro-cli's built-in steering (Module 4).

MCP in plain English

If the harness is the agent's body and tools are its hands, MCP is the standard plug it uses to connect new hands. Without MCP, every tool integration is bespoke. With MCP, your team writes one server and every MCP-aware agent can use it.

3.1 The problem MCP solves

Before MCP, every agent ↔ tool integration was custom:

  • Agent A talks to GitHub via Python SDK with one auth pattern
  • Agent B talks to GitHub via REST with a different auth pattern
  • Agent C uses a third-party wrapper
  • None of them share code

Every new integration meant re-implementing auth, error handling, schema definitions, rate-limit handling. It didn't scale, and it locked you into one agent vendor's tool ecosystem.

3.2 What MCP is

Model Context Protocol is an open spec — a client/server protocol that standardizes three primitives:

PrimitiveWhat it isExample
ToolsActions the agent can invokecreate_issue, query_database, deploy_lambda
ResourcesData the agent can readA file, a record, a doc URL
PromptsPre-built templates the user/agent can invoke"Summarize PR <id>" with bound args

An MCP server exposes some combination of these. An MCP client (the agent harness — kiro-cli) connects to servers and makes their tools/resources/prompts available to the LLM.

HTTP standardized how clients talk to servers on the web. Any browser works with any compliant web server. MCP does the same for agent ↔ tool: write one MCP server, and any MCP-aware agent (kiro-cli, Claude Code, custom agents) can use it.

3.3 The architecture

+---------------+      stdio / HTTP       +-----------------+
|   kiro-cli    |  <-------------------   |   MCP server    |
|  (MCP client) |     JSON-RPC msgs       | (GitHub, DB,    |
|               |                         |  AWS, custom)   |
+---------------+                         +-----------------+
       |                                         |
       v                                         v
   LLM sees tools                          Executes tools
   from this server                        against the
   in its tool list                        real system

Two transports:

  • stdio — the server runs as a local subprocess; messages over stdin/stdout. Best for local dev tools.
  • SSE / HTTP — the server runs as a remote service. Best for shared team servers.

The wire protocol is JSON-RPC. You almost never write it by hand — SDKs exist for Python, TypeScript, Go.

3.4 How kiro-cli consumes MCP

kiro-cli reads MCP server configs from JSON files. Two scopes:

ScopePathWhen to use
Global~/.aws/kiro-cli/mcp.jsonPersonal servers across all projects
Workspace.kiro/settings/mcp.jsonProject-shared, committed to the repo

Example workspace config:

{
  "mcpServers": {
    "github": {
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-github"],
      "env": { "GITHUB_TOKEN": "${GITHUB_TOKEN}" }
    },
    "postgres-local": {
      "command": "uvx",
      "args": ["mcp-server-postgres", "postgresql://localhost/myapp"]
    }
  }
}

When kiro-cli starts, it launches each configured server as a subprocess (stdio) or connects over HTTP. The server reports its tools. kiro-cli adds them to the LLM's tool list with a namespace prefix like github___create_issue.

Per-agent scoping (preview — Module 6):

{
  "name": "db-investigator",
  "mcpServers": ["postgres-local"],
  "tools": ["fs_read"]
}

This agent has access to only the Postgres MCP server's tools plus fs_read. Principle of least privilege.

3.5 What MCP isn't

Three misconceptions to clear up for your team:

  • MCP is not a model. It's a protocol. Models are still Claude, GPT, etc.
  • MCP is not an agent framework. It doesn't define how agents think or loop. That's the harness's job.
  • MCP is not magic. Each MCP server is code someone wrote — auth, error handling, and bugs included. Vet servers like any other dependency.
MCP is plumbing. It standardizes the socket, not what's on the other end.

3.6 When to use MCP vs built-in tools

kiro-cli already has fs_read, fs_write, execute_bash, web_search as built-ins. Why not put everything in MCP?

Use built-inUse MCP
Filesystem, shell, web fetch — universal needsAnything domain-specific (GitHub, AWS, your DB)
Performance-critical pathsAnything needing auth flows or credentials
Things the harness needs for safety (allowedPaths)Anything you want hot-swappable across agents

Rule of thumb: if multiple agents (or projects) would benefit from the same integration, write it as an MCP server. If it's a one-off, a direct tool or execute_bash call is fine.

3.7 Servers worth knowing

A rough survey (verify currency before adopting — the ecosystem moves fast):

  • Official — filesystem, git, github, gitlab, postgres, slack, brave-search, fetch (@modelcontextprotocol/server-* packages)
  • AWS / cloud — community servers for S3, Lambda, RDS, etc.
  • Internal — write your own for company-specific systems (deploy pipelines, internal APIs, runbooks)

Writing your own is straightforward — TypeScript or Python SDK, define tool schemas, implement handlers. A useful first one for your team: an MCP server that wraps your internal docs/wiki search.

Wire up an MCP server

Add the official filesystem MCP server to kiro-cli scoped to a sandbox directory.

Add to ~/.aws/kiro-cli/mcp.json:

{
  "mcpServers": {
    "fs-sandbox": {
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-filesystem", "/tmp/kiro-sandbox"]
    }
  }
}

Then in kiro-cli, ask:

Use the fs-sandbox MCP server to list files in the sandbox and create
a hello.txt with the current timestamp.

Inspect the tool calls. You should see fs_sandbox___list_directory and fs_sandbox___write_file (or similar). Notice the namespace prefix — that is how kiro-cli disambiguates MCP tools from built-ins.

Also try /mcp in interactive mode to see connected servers and their tools.

Steering the agent

Steering is the first kiro-cli primitive. It is the answer to "how do I stop telling the agent my conventions on every turn?" — and it is the foundation for every other primitive in Part 2.

4.1 The problem steering solves

In Part 1 we saw the LLM is stateless. Every turn starts blank. If you want it to:

  • Use your team's commit message format
  • Prefer Terraform over CDK
  • Never edit files under infra/prod/
  • Always run pytest after Python edits
  • Follow your CSS naming convention

...you'd have to write that into every prompt. That doesn't scale and it leaks across team members.

Steering files solve this. They are markdown files in your repo that kiro-cli automatically includes in the model's context based on rules you define.

Steering is durable system-prompt content, version-controlled with your repo, scoped by file glob.
Mental Model

Middleware that runs before every model call, injecting the relevant rules so the model doesn't start blank.

4.2 Where steering files live

your-repo/
├── AGENTS.md              # Always-loaded project context (top-level)
├── .kiro/
│   └── steering/
│       ├── conventions.md       # Inclusion mode set per file
│       ├── aws-patterns.md
│       └── security-rules.md
└── src/

Two kinds:

AGENTS.md (top-level) — Always loaded for every kiro-cli session in this workspace. Your "the agent must always know this" file. Keep it short — every token here costs context budget on every turn.

.kiro/steering/*.md — Granular steering files with three possible inclusion modes, declared per file:

ModeWhen it loadsUse case
alwaysEvery turn (like AGENTS.md but separable)Core team rules, always relevant
fileMatchWhen files matching a glob enter contextTech-specific rules: "when editing *.tf, prefer modules"
manualOnly when @-referenced in chatNiche playbooks, deep-dive guides

The mode is set in YAML front-matter at the top of each file.

4.3 Anatomy of a steering file

---
inclusion: fileMatch
fileMatchPattern: "**/*.tf"
---

# Terraform conventions for this repo

- Use modules from `modules/` — don't inline resources except for one-offs
- Stack naming: `<env>-<service>-<resource>` (e.g., `prod-api-rds`)
- Every resource gets a `Project` and `Owner` tag
- State backend is S3 in `terraform-state-<account-id>`
- Never use the `local-exec` provisioner. Use a `null_resource` with a
  signed payload if absolutely necessary.

## When asked to add a new resource

1. Check if a module already exists in `modules/`
2. If not, propose creating one with reusability in mind
3. Default to least-privilege IAM

Front-matter is YAML. Everything below is markdown the model reads as additional system instructions on the turns where this file is loaded.

4.4 Inclusion modes in detail

always — loaded every turn, regardless of context.

  • Best for: team identity, hard "never do this" rules, critical security boundaries
  • Cost: every always-loaded file consumes tokens on every turn. Keep these lean.

fileMatch — loaded when a file matching the glob appears in the active context (you mention it, the agent reads it).

  • Best for: language conventions (**/*.py), infra rules (infra/**/*.tf), stack-specific patterns
  • Cheap — only consumes tokens when relevant

manual — loaded only when the user explicitly @-references it in chat:

Continue with the deployment runbook. @runbook
  • Best for: playbooks too long to always include, role-specific workflows, onboarding content

4.5 What goes in steering vs everywhere else

Common mistake: put everything in steering. It bloats context, slows responses, and turns AGENTS.md into a CV.

Goes in steeringGoes elsewhere
Conventions, style, "always do X" rulesOne-off task instructions → just put them in the prompt
Project-wide architectural decisionsDetailed runbooks for specific tasks → manual steering or skills (Module 5)
Security boundariesMechanically-enforceable rules → hooks (Module 8)
Tech-stack patternsCode that should be reusable → just write the code
If a rule must hold across every interaction in this repo, it goes in always-loaded steering. Everything else has a better home.

4.6 A real-shape AGENTS.md

# Platform team agent context

## Stack
- Python 3.12, FastAPI, Pydantic v2
- Infra: Terraform on AWS (eu-west-1 primary)
- CI: GitLab CI
- Tests: pytest, no other framework
- Lint: ruff (line length 100), mypy strict

## Repo layout
- `src/` — application code
- `infra/` — Terraform modules and environments
- `tests/` — pytest tests, mirrors `src/` layout
- `.gitlab/` — CI pipelines

## Behavior rules

- Never write to `infra/prod/` without an explicit
  "I want to touch prod" from the user

## When unsure

Short, scannable, no fluff. Every line earns its tokens. This file is loaded on every turn — treat it like an SLO, not a wiki.

Skills: invokable workflows

Steering tells the agent the rules. Skills give it the playbooks. A skill is a named, reusable workflow registered as a slash command — version-controlled, scoped to your repo, and shareable across the team.

5.1 What a kiro-cli skill is

A skill is a directory containing a SKILL.md (plus any companion files) that bundles instructions, references, and templates into one named workflow. On kiro-cli startup, each skill auto-registers as a slash command in the TUI.

.kiro/skills/
└── security-review/
    ├── SKILL.md
    ├── checklist.md
    └── examples/
        └── good-iam.tf

The user types /security-review. The agent loads SKILL.md, follows its instructions, and produces the expected output.

A skill is a runbook the agent can read and act on, exposed as a slash command.

5.2 Anatomy

SKILL.md:

---
name: security-review
description: Audit a Terraform change for IAM least-privilege, public
  exposure, and secret handling. Use before any apply to non-dev environments.
---

# Security review

You are running a security review on a Terraform change.

## Inputs needed
- The path to the Terraform module or changed files
- The target environment (dev / staging / prod)

## Steps
1. Read the changed `.tf` files.
2. For each `aws_iam_*` resource: flag wildcards in actions/resources.
3. For each public resource (`aws_lb`, `aws_s3_bucket`, ...): confirm an
   explicit access rule; flag unintentional `0.0.0.0/0` ingress.
4. Grep for hardcoded secrets in `.tf` files.
5. Produce a markdown report with one section per finding (severity,
   file, line, suggested fix).

## Output format
Markdown with `## Finding N` headers. Do not modify files — review only.

Two things matter:

  • name is the slash command. Keep it short.
  • description is the elevator pitch. The agent uses it to decide relevance when the user doesn't invoke explicitly.

5.3 Skills vs steering

  • Convention or constraint → steering

5.5 Writing a good skill

Three things matter:

  1. The name is the slash command. Short and memorable. /sec-review beats /security-review-checklist-v2.
  2. The description helps the agent recognise relevance. Even with explicit invocation, the agent may propose a skill when context matches. Description = elevator pitch.
  3. The body should be runnable. Step-by-step, with checks and decision points. Not philosophy — instructions the agent will follow line by line.

Agents: scoped runtimes

Steering says what the agent should know. Skills say what the agent should do. An agent config says what the agent is allowed to do — model, tools, MCP servers, file boundaries, persona — wrapped in a JSON file. This is where least-privilege starts to bite.

6.1 Why custom agents

The default agent (kiro_default) has access to everything: all built-in tools, all your MCP servers, no path restrictions. Great for exploration. Dangerous for repeatable workflows.

Custom agents are how you build constrained, special-purpose assistants on top of the same harness. Examples:

  • A code reviewer that can read but not write
  • A DB investigator with only the read-only Postgres MCP and nothing else
  • A doc updater that can only touch .md files
  • A production-deploy assistant scoped to infra/prod/ and gated by hooks
An agent is a constrained version of kiro-cli, persisted as a config file, named, and switchable.

6.2 Anatomy of an agent config

File location:

  • Workspace — .kiro/agents/<name>.json (committed to the repo)
  • Global — ~/.aws/kiro-cli/agents/<name>.json (personal)

A representative shape (exact field names may evolve — check the version you have installed):

{
  "name": "code-reviewer",
  "description": "Read-only code review against repo conventions",
  "model": "claude-sonnet-4",
  "prompt": "You are a senior code reviewer. Read changed files,
            run linters, produce a markdown review. Never modify files.",
  "tools": ["fs_read", "execute_bash"],
  "allowedTools": {
    "execute_bash": ["pytest *", "ruff check *", "mypy *"]
  },
  "resources": [
    "file://./AGENTS.md",
    "file://./.kiro/steering/*.md"
  ],
  "mcpServers": []
}

6.3 Key fields

FieldPurpose

6.4 Switching agents

In the TUI:

/agent              # list available agents
/agent code-reviewer

Headless mode:

The agent is fixed for the duration of a session. You don't hot-swap mid-conversation; you start a new session with a different agent.

6.5 Least privilege: a real example

The bad way — using the default agent for sensitive work:

kiro-cli "Investigate why the Postgres connection pool is exhausted in prod"

Default agent has fs_write, unrestricted execute_bash, every MCP server. The agent might decide to "fix" something by editing prod config. Not great.

The better way — a db-investigator agent:

{
  "name": "db-investigator",
  "description": "Read-only Postgres investigation; reports findings, never modifies",
  "model": "claude-sonnet-4",
  "prompt": "You investigate database issues by querying Postgres (read-only)
            and reading code. You never write to the database or modify
            files. End with a findings report.",
  "tools": ["fs_read", "execute_bash"],
  "allowedTools": {
    "execute_bash": ["grep *", "ls *", "cat *"]
  },
  "mcpServers": ["postgres-prod-readonly"],
  "resources": [
    "file://./AGENTS.md",
    "file://./docs/db-runbooks/*.md"
  ]
}

Run:

kiro-cli --agent db-investigator "Why is the connection pool exhausted?"

The agent can read code, query the read-only DB MCP, run safe bash. It cannot write files, run arbitrary commands, or touch any other MCP. The blast radius is bounded by the config, not by trust in the model.

6.6 Steering, skills, agents — the triangulation

Three primitives, three jobs. They compose: a custom agent loads its own resources/steering, can invoke skills as slash commands, and is bounded by its tools/paths config. Describe with steering, restrict with agents, encode know-how in skills.

6.7 Patterns for agent design

  1. Read-only investigator — security audits, perf debugging, root-cause work. Tools: fs_read, execute_bash (safe-only), read-only MCPs. No fs_write.
  2. Scoped writer — doc updaters, test generators. fs_write is allowed but allowedPaths restricts the blast radius.
  3. Single-purpose builder — for one task ("generate Pydantic models from OpenAPI", "convert REST to GraphQL"). Locked tools, prompt-as-spec.
  4. Coordinator — high-level prompt + the subagent tool to delegate to child agents (preview of Module 7).

Multi-agent & subagents

One agent can do a lot. But when a task fans out — parallel investigations, isolated dangerous operations, multi-perspective reviews — you want more than one. kiro-cli's subagent tool lets a parent spawn child agents with their own configs, each fully scoped. There is no /spawn command. This happens inside the loop.

7.1 The subagent tool

Clear this up first: there is no /spawn slash command in kiro-cli. Subagent spawning is a tool call the LLM makes, just like fs_read or execute_bash. From the LLM's perspective:

{
  "tool": "subagent",
  "input": {
    "agent": "code-reviewer",
    "prompt": "Review the changes in src/auth/"
  }
}

The harness sees this, spins up a new kiro-cli runtime configured as code-reviewer, runs it with the given prompt, captures its final response, and returns that to the parent as the tool result. The parent continues with that result in context.

A subagent is a recursive kiro-cli session, scoped to a child agent config, with the result piped back to the parent.

7.2 Anatomy of an orchestrator

The parent (orchestrator) is a regular agent with the subagent tool enabled and an availableAgents whitelist:

{
  "name": "review-orchestrator",
  "description": "Coordinate a multi-perspective review",
  "model": "claude-sonnet-4",
  "prompt": "You coordinate reviews. For a given MR, spawn a code-reviewer
            subagent for code quality, a security-reviewer for security,
            and a doc-reviewer for docs. Combine findings into one report.
            You don't read files directly — delegate.",
  "tools": ["subagent"],
  "availableAgents": ["code-reviewer", "security-reviewer", "doc-reviewer"],
  "trustedAgents": ["doc-reviewer"]
}

Three new fields:

  • tools: ["subagent"] — the parent has only this tool. It can delegate but not act directly. (Add fs_read if it should read for context.)
  • availableAgents — whitelist of child agents this parent may spawn
  • trustedAgents — subset that skip the permission prompt when spawned

7.3 The orchestrator pattern

The pattern that's most useful in practice:

  • Parent: read-only or no-fs. It plans, delegates, summarises. Cheap model (Haiku) often suffices.
  • Children: scoped writers or investigators. Each has its own allowed tools, MCP servers, and file boundaries.

What you gain:

  • Bounded blast radius — only children can write, each in its own scope
  • Parallelism — parent can spawn independent children concurrently
  • Token efficiency — children's full reasoning doesn't pollute parent context; only their final output returns

7.4 DAG planning

            review-orchestrator
                    |
        +-----------+-----------+
        |           |           |
   code-rev     sec-rev     doc-rev
                    |
                synthesise

The parent decides which children are independent (parallel) and which have dependencies (sequential). It composes the result at the end. Fork/join applied to LLM workflows. The parent is the join point.

7.5 What subagents CAN'T do

Constraints to internalise before you architect around subagents:

ConstraintImplication
Hooks don't fire on subagentsA preToolUse hook only runs on the main agent. Subagent writes are not gated by hooks. Lock down at the agent config instead.
Typically no MCP / web tools in subagentsMany MCP servers can't be safely shared. Verify per-server before assuming.
No nested orchestration by defaultA subagent typically cannot spawn its own subagents (or only one level, depending on config).
Their context is freshA subagent does not see the parent's conversation. You must pass everything in the prompt.
You only get the final responseThe parent doesn't see the subagent's tool calls, intermediate reasoning, or files read. Just the final text.

The last two combined matter most: the prompt to a subagent is a complete brief, not a conversation.

7.6 Inspecting subagents at runtime

In the TUI, press Ctrl+G to open the subagent monitor — see spawned children, their status, their progress reports. Useful when an orchestration is opaque from outside.

7.7 When NOT to use subagents

SituationBetter alternative
Single linear taskOne agent
Task needs fine-grained hook enforcementMain agent (hooks don't fire on subs)
You want to see every step turn-by-turnMain agent (subagents are opaque between start/finish)
All operations share the same trust levelOne scoped agent

Subagents shine when you have parallelism + scope isolation as joint requirements. With only one of those, simpler structures work.

7.8 A concrete fan-out

User runs:

kiro-cli --agent review-orchestrator "Review MR !1234"

Inside the loop:

  1. Parent calls subagent(code-reviewer, "Review src/ changes in branch X") → waits
  2. Parent calls subagent(security-reviewer, "Audit IAM changes in branch X") → waits
  3. Parent calls subagent(doc-reviewer, "Check docs match changes in branch X") → waits
  4. Parent receives three markdown reports
  5. Parent synthesises into one final report

If the parent is smart and the harness supports it, calls 1-3 can run in parallel. The whole orchestration completes in roughly max(child durations) instead of sum.

Hooks: mechanical enforcement

Steering is advisory. Agents are scoped. Hooks are mechanical enforcement. They are bash (or any executable) that runs at lifecycle events. If you need to actually block an action — not just ask the LLM nicely — you need a hook.

8.1 The lifecycle

kiro-cli fires events around the agent loop. Hooks are scripts you register against these events.

EventWhen it firesCan it block?
agentSpawnA session/agent startsNo (initialisation only)
userPromptSubmitThe user submits a promptYes (exit non-zero to reject)
preToolUseThe LLM is about to call a toolYes (exit code 2 blocks the call)
postToolUseA tool has returnedNo (side effects only)
stopThe agent is about to return final textYes (can force another turn)
preToolUse exit code 2 is the only mechanism that stops the LLM from doing something. Everything else either prepares or observes.

8.2 Where hooks live

Workspace-scoped, committed with the repo:

.kiro/hooks/
├── pre-tool-use.sh
├── post-tool-use.sh
└── stop.sh

Each script receives event data on stdin (JSON) and can return JSON on stdout to influence behavior. Exit code matters per event.

8.3 Anatomy: block writes to infra/prod

.kiro/hooks/pre-tool-use.sh:

#!/usr/bin/env bash
set -euo pipefail

# Read event payload from stdin
event=$(cat)

tool=$(echo "$event" | jq -r '.tool')
path=$(echo "$event" | jq -r '.input.path // empty')

# Block fs_write or execute_bash that touches infra/prod/
if [[ "$tool" == "fs_write" && "$path" == infra/prod/* ]]; then
  echo '{"reason": "Direct writes to infra/prod/ are blocked. Use the prod-deploy spec."}'
  exit 2   # exit 2 = block the tool call
fi

exit 0   # allow

Make it executable. From now on, any attempt by the LLM to write under infra/prod/ is hard-blocked. The model sees the rejection reason and adapts.

8.4 Hooks vs steering vs agent config

MechanismStrengthWeakness
SteeringEasy to write; advisoryLLM can ignore it
Agent config (tools / allowedPaths)Enforced by harness; declarativeCoarse-grained; one config per session
HooksProgrammatic; can inspect content; can block at exact decision pointsOperational overhead; needs bash + jq skill in the team

Use hooks when you need content-aware enforcement — e.g., "block fs_write to files containing the literal secret=" — which neither steering nor static config can express.

8.5 Useful hook patterns

8.6 Hooks only fire on the main agent

This was the trap from Module 7 — worth repeating:

Hooks do not fire on subagents. If subagent enforcement matters, do it via the subagent's own config (tools whitelist, allowedPaths, prompt).

If you need belt-and-braces enforcement, combine:

  • Hook on parent (blocks main-agent dangerous tool use)
  • Restrictive agent config on each child (blocks the same things at the runtime level)

Both layers fail closed independently.

Specs: gated builds

For non-trivial work — anything beyond "tweak this function" — you want explicit alignment before code. Specs give you a three-stage, human-gated build: requirements → design → tasks. The output is a directory of markdown files the agent then implements.

9.1 When to use a spec

Use a spec when:

  • The task touches more than 2-3 files
  • Requirements are unclear and need to be teased out
  • You want a record of what was decided before code existed
  • The work spans multiple sessions (you need persistent state)
  • Stakeholders need to review the plan before execution

Skip the spec when:

  • The task is a clear one-shot ("rename this variable everywhere")
  • You already know the design
  • The work fits in one conversation
Specs trade speed for clarity. For agentic work, the clarity is usually worth it.

9.2 The three documents

A spec is a directory:

.kiro/specs/<feature-name>/
├── requirements.md   # What we're building, in EARS notation
├── design.md         # How we'll build it
└── tasks.md          # The implementation breakdown

Each document is built in order, with a human-in-the-loop gate before moving to the next.

StageWhat it containsGate
RequirementsEARS-format requirements, acceptance criteriaUser reviews + approves before design
DesignArchitecture, data model, interfaces, sequenceUser reviews + approves before tasks
TasksNumbered, parallelisable, atomic tasksUser can edit before execution

9.3 EARS notation

Easy Approach to Requirements Syntax. Five canonical patterns that force unambiguous requirements:

PatternFormExample
Ubiquitous"The system shall X"The system shall log every authentication attempt.
Event-driven"When X, the system shall Y"When a user submits an invalid token, the system shall return HTTP 401.
State-driven"While X, the system shall Y"While in maintenance mode, the system shall return HTTP 503.
Conditional"If X, then the system shall Y"If the rate-limit is exceeded, then the system shall return HTTP 429.
Optional"Where X, the system shall Y"Where multi-tenant is enabled, the system shall scope all queries by tenant_id.

EARS feels stilted at first. That's the point — ambiguity in requirements becomes bugs in code. Forcing the structure exposes the ambiguity before code.

9.4 Worked example: requirements.md

# Requirements: API Key Rotation

## REQ-1: Key generation
The system shall generate new API keys using 256-bit random tokens.

## REQ-2: Active key rotation
When an admin triggers rotation for a tenant, the system shall:
- Generate a new key marked `active`
- Mark the previous key `deprecated` with a 30-day expiry
- Emit a `key.rotated` event to the audit log

## REQ-3: Deprecated key acceptance
While a key is in `deprecated` state and within its expiry window, the system
shall accept it but return a `X-Key-Deprecated` response header.

## REQ-4: Expiry enforcement
If a key has expired, the system shall return HTTP 401 with body
`{"error": "key_expired"}`.

## Out of scope
- Per-endpoint key restrictions (future work)
- Programmatic rotation by tenant (admin only for v1)

A senior engineer can sanity-check this in two minutes. The agent has zero room to invent behavior.

9.5 design.md

After requirements approval, the agent produces a design:

# Design: API Key Rotation

## Data model
- `api_keys` table:
  - `id` (uuid), `tenant_id` (uuid), `hash` (sha256), `status` (enum: active|deprecated|revoked),
    `expires_at` (timestamp nullable), `created_at`, `rotated_from_id` (nullable)

## API surface
- `POST /admin/tenants/{tenant_id}/keys/rotate` — admin endpoint
- Existing middleware: `verify_key()` updated to handle `deprecated` status

## Sequence (rotation)
1. Admin POST hits the rotate endpoint
2. Service generates new token, hashes, inserts as `active`
3. Existing `active` row updated to `deprecated`, `expires_at = now + 30d`
4. `key.rotated` event published to SNS
5. Response returns the new plaintext token (one-time)

## Tradeoffs considered
- Soft delete vs hard delete: soft delete chosen for audit trail
- 30-day expiry chosen to match SOC2 requirements

User reviews. If the data-model or API choice is wrong, this is the cheapest moment to catch it — before any code.

9.6 tasks.md

# Tasks: API Key Rotation

- [ ] T1: Create migration for `api_keys` schema changes
- [ ] T2: Update `ApiKey` Pydantic model with new fields
- [ ] T3: Update `verify_key()` middleware to handle `deprecated` status
- [ ] T4: Add `POST /admin/tenants/{tenant_id}/keys/rotate` endpoint
- [ ] T5: Publish `key.rotated` event to SNS topic `audit-events`
- [ ] T6: Tests: rotation flow, deprecated key acceptance, expiry enforcement
- [ ] T7: Update docs: `docs/api/auth.md`

Dependencies: T2 depends on T1. T3-T5 can run in parallel after T2.

The agent (or a fan-out orchestrator) implements one task at a time, marking it done, running tests. The tasks list is your durable progress tracker.

9.8 Patterns

  • Spec for feature, skill for procedure. A spec builds a feature once; a skill runs a procedure repeatedly.
  • Specs as RFC. Treat requirements.md and design.md like internal RFCs — review them as artefacts, not just intermediate agent output.
  • Tasks become MR titles. Each task is small enough to be one MR. Helps with review.
  • Pause between stages. Don't let the agent rush all three in one turn. The gate is the point.

Context: the scarce resource

Context is the agent's working memory. It's finite. It's expensive. It decays as conversations grow. Managing it deliberately is the single most important operational skill in agentic coding — more than prompt-crafting, more than tool selection.

10.1 What's in context every turn

On every model call, kiro-cli assembles a payload that typically includes:

  • System prompt (agent-specific from the config)
  • Always-loaded steering — AGENTS.md + inclusion: always files
  • Conditionally-loaded steering — fileMatch-glob files that became relevant
  • Resources — anything in the agent's resources array
  • Conversation history — every prior user message, assistant message, tool call, tool result
  • The current user prompt

All of this counts against the context window. Frontier models top out at ~200K tokens. By turn 50 of a real session, you can easily hit 100K+ just from accumulated history.

10.2 Inspecting context

In the TUI:

/context

Shows what's currently loaded — steering files, resources, token counts. Use this. Often. Most "why is the agent ignoring my rule?" problems are revealed by /context showing the rule wasn't loaded.

Other useful commands (Module 12 deep-dive):

  • /tools — what tools this session has
  • /mcp — which MCP servers are connected
  • /compact — summarise history to free tokens (loses fidelity)
  • /clear — wipe conversation, keep agent + steering

10.3 The three context budgets

Mental model: you're managing three competing budgets.

BudgetDriven byIf you exceed
Window (~200K tokens)Total context sizeRequest fails or truncates silently
Cost (latency + $)Per-turn token countSlow, expensive turns
Signal (attention)Relevance of what's in contextModel loses focus, ignores rules, hallucinates

The signal budget is the sneaky one. Even if you fit in the window, a context full of stale tool results is noisier than a smaller, curated one. The model attends less to instructions buried under 50K tokens of old fs_read output.

10.4 Context decay strategies

Four moves, ranked by cost:

  1. /clear — start fresh. Cheapest move. Keep agent + steering, drop history. Use when one task is done and the next is unrelated.
  2. /compact — summarise history into a smaller version. Mid-cost. Use when you need to keep going but history has gotten bloated. Beware: details get lost.
  3. New session entirely — exit and restart. Forces full reset. Use after long sessions where state has accumulated unpredictably.
  4. Spec-driven multi-session work — for long-running features, drop spec docs (Module 9) and pick up later. Spec files preserve state where chat can't.

10.5 What inflates context the fastest

SourceTypical bloat
Tool results — file reads on large files5K-50K tokens per call
Conversation history (every turn repeats)Linear growth
Many always-loaded steering filesConstant per-turn cost
Verbose system promptsConstant per-turn cost
MCP tool resultsVariable; can be large for DB query results

Defensive habits:

  • Prefer grep and targeted reads over cating entire files
  • Keep AGENTS.md short
  • Move bulk content to fileMatch steering or manual skills
  • For long sessions, periodically /compact or /clear

10.6 Patterns that work

  • One session per task. Don't run "fix the auth bug" and "add a new endpoint" in the same session.
  • Read narrowly. Before letting the agent fs_read a 5K-line file, ask if it can grep for the symbol first.
  • Cite, don't re-read. Tell the agent "you read auth.py earlier; assume that still holds" rather than letting it re-read.
  • Push state into files, not chat. Notes, decisions, intermediate findings — write them to a file the agent can re-read later.
  • Subagents for noisy work. When investigation will produce a lot of tool calls, spawn a subagent and let it return a summary (Module 7).

10.7 Common failure modes

Replies get slower over timeHistory accumulating/compact or /clear

Orchestration & planning

The non-deterministic engine works best on small, well-scoped pieces. The skill is structuring real work so it decomposes into pieces the agent can ship reliably. That's orchestration.

11.1 The research → plan → implement loop

For anything beyond a one-line change, run this three-pass pattern in kiro-cli:

  1. Research — agent reads the relevant files, runs grep, queries MCP servers. Goal: ground the model in the actual state of the code. Output: a written summary of what exists today.
  2. Plan — agent proposes an approach. Files to change, order of changes, risks. Output: a numbered plan. You review.
  3. Implement — agent executes the plan, one step at a time, running tests after each.

The gate between 2 and 3 is where you catch most disasters. The cost of a bad plan corrected in 2 is one minute. The cost of a bad plan implemented in 3 is hours.

11.2 Why not "just do it"

It's tempting to say "fix this bug" and let the agent loose. Why structure?

Without research → plan → implementWith it
Agent invents details that don't match realityGrounded in real code
You discover bad approach after files are changedCatch in plan phase
Long opaque tool-call streamReviewable artefact at each stage
Hard to resume after interruptionPlan is a re-entry point
Token waste on speculative readingTargeted reads

11.3 How to invoke the pattern in practice

You don't need a primitive for this — a steering rule and a prompt convention is enough.

AGENTS.md snippet:

## Working style
For any task touching more than 2 files:
1. First, summarise what you've read and what currently exists. Stop and confirm.
2. Then propose a numbered plan. Stop and confirm.
3. Only then implement, one step at a time, running tests after each.
Do not skip steps. Do not combine them.

For larger work, use a spec (Module 9). For one-off-but-careful work, this lighter pattern works.

11.4 TODO lists and progress reporting

For multi-step implementations, ask the agent to maintain an explicit TODO list:

Implement the API key rotation feature. Maintain a TODO list in this conversation,
mark items as you complete them, and call report_progress between items.

Two benefits:

  • Recoverable. If context decays or session restarts, the TODO is the resume point.
  • Visible. You can see at a glance what's done and what isn't.

The report_progress built-in tool surfaces status to the TUI — useful for long-running tasks.

11.5 Subagent fan-out for parallel orchestration

When research or implementation has independent branches, use subagents (Module 7):

  • Research fan-out — parent dispatches subagents to investigate different subsystems in parallel; collects summaries.
  • Implementation fan-out — after plan approval, parent dispatches each task to a scoped writer subagent.

Caveat: only use fan-out when work is genuinely independent. Sequential tasks with state dependencies are simpler in one agent.

11.6 When orchestration becomes overhead

Don't apply this pattern when:

  • The task is trivial (one-liner, obvious change)
  • You're exploring or prototyping (research-as-you-go is fine)
  • The agent will produce a discardable artefact (a one-shot draft)

Match the discipline to the stakes.

Best practices and failure modes

A synthesis. The patterns that distinguish teams who get value from kiro-cli from teams who fight it. Plus the LICENSE gotcha that's caught more than one team off-guard.

13.1 The mantra

Describe with steering. Restrict with agents. Encode know-how in skills.

Three sentences. Internalise them. Every kiro-cli design decision should be testable against this.

  • A team rule? → steering
  • A repeatable procedure? → skill
  • A bounded runtime for a risky job? → agent

When in doubt, default to the lightest primitive that fits.

13.2 Treat .kiro/ as code

.kiro/ files are infrastructure. Treat them accordingly:

  • Commit them. Steering, skills, agents, hooks, specs — all in the repo, reviewed in MRs.
  • Lint them. Validate JSON configs, check YAML front-matter, ensure hook scripts are executable.
  • Test them. When you change a hook, run kiro-cli with a known prompt and assert behavior.
  • Version them. Pin kiro-cli versions in CI; upgrade deliberately; note breaking changes.
  • Document them. A short README.md in .kiro/ explaining what each piece does.

The team that doesn't do this ends up with mystery configs no one can explain six months later.

13.4 When NOT to use kiro-cli

Honest list:

  • Deterministic refactors. A rename, a regex substitution, a codemod. Use a real refactor tool.
  • Code that must be exactly reproducible. Generated migrations, build artefacts. The non-determinism bites.
  • Hot-path performance work. The agent loop adds latency. Profile manually, edit manually.
  • Greenfield architectural decisions. The agent has no taste. Decide first, implement second.
  • Anywhere a wrong answer is catastrophic and undetectable. Money calculations without tests, security-critical crypto.

Knowing where the tool doesn't fit makes you better at recognising where it does.

13.5 LICENSE gotcha

Important — LICENSE

kiro-cli's LICENSE forbids use in conjunction with OpenClaw, NemoClaw, and similar reverse-engineered or Anthropic-API-fronting clients.

This matters if your team uses (or experiments with) those tools. Concretely:

  • Do not mix kiro-cli configs/skills with OpenClaw/NemoClaw setups
  • Do not pipe kiro-cli output into one of those clients, or vice versa
  • If a project uses both, separate the workflows entirely

Audit your team's stack. If anyone is doing this, fix it before it becomes a procurement issue.

13.6 Habits of effective teams

Observed in teams that ship reliably with kiro-cli:

  • One scoped agent per risky workflow. Never use default for prod-adjacent work.
  • AGENTS.md under 100 lines. Anything more, move to fileMatch.
  • A code-reviewer agent that runs in CI on every MR. Cheap, useful; agent-output is reviewed by humans regardless.
  • Periodic .kiro/ review. A 30-minute monthly grooming session pays back tenfold.
  • Specs for anything beyond a 2-file change. Cheap discipline; massive savings.
  • Templates for new agents and skills. Lowers the activation energy to do things properly.

  echo '{"reason": ".env files contain secrets. Edit manually."}'
  exit 2
fi

exit 0

      kiro-cli --no-interactive --agent code-reviewer \
        "Review the diff between $CI_MERGE_REQUEST_DIFF_BASE_SHA and HEAD.
         Focus on the Findings section. Output strict markdown to stdout." \
        > review.md
    - |
      # Post review as MR comment
      curl --request POST \
        --header "PRIVATE-TOKEN: $GITLAB_TOKEN" \
        --form "body=<review.md" \
        "$CI_API_V4_URL/projects/$CI_PROJECT_ID/merge_requests/$CI_MERGE_REQUEST_IID/notes"
  artifacts:
    paths: [review.md]
    when: always

Verify:

  1. Open a small MR on your branch
  2. Watch the agent-review job run
  3. Confirm review.md is uploaded as an artifact
  4. Confirm a review comment appears on the MR

What you learned: Headless mode + agent config + CI = automated review on every MR. The same code-reviewer.json runs locally and in CI, with the same conventions. (Module 12.)
